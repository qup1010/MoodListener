# 📱 MoodListener 修复验证与测试指南

本文档提供详细的步骤，帮助您验证最近的修复内容（暗黑模式、位置服务、UI适配等），并确保应用在 Android 设备上运行正常。

## 🛠️ 第一步：构建与部署

在开始测试前，请确保将最新的代码构建并同步到 Android 项目中。

### 1. 构建 Web 资源
```bash
npm run build
```

### 2. 同步到 Android 平台
```bash
npx cap sync android
```

### 3. 打开 Android Studio 并运行
```bash
npx cap open android
```
*在 Android Studio 中点击 "Run" (绿色的播放按钮) 将应用安装到模拟器或真机。*

## ✅ 第二步：功能验证清单

请按照以下顺序逐一测试修复的功能。

### 1. 🌓 暗黑模式启动修复 (Dark Mode Flash Fix)
**目标**：确保应用启动时根据系统或设置正确应用主题，没有白色闪烁。
*   **操作**：
    1.  在手机系统设置中开启「深色模式」。
    2.  完全关闭 MoodListener 应用（从后台划掉）。
    3.  重新打开应用。
*   **预期结果**：
    *   启动画面（Splash Screen）后，应用直接显示深色背景。
    *   **不会**先显示白色背景再突然变黑。

### 2. 📍 位置服务与 API Key (Location & Amap)
**目标**：验证高德地图 API Key 的配置及位置服务的准确性。
*   **测试场景 A：默认/无 Key 状态**
    1.  进入「设置」 -> 「位置服务设置」。
    2.  确保输入框为空，点击保存。
    3.  回到首页，点击「每日签到」进入记录页。
    4.  点击位置图标。
    *   **预期**：应该显示基于 BigDataCloud 或 GPS 坐标的粗略位置，或弹出手动输入提示（如果 API 访问受限）。**不再有烦人的 blocking alert**。
*   **测试场景 B：配置高德 Key**
    1.  进入「设置」 -> 「位置服务设置」。
    2.  输入您的高德 Web 服务 API Key。
    3.  点击「保存配置」，确认提示成功。
    4.  回到记录页，点击位置图标。
    *   **预期**：应能迅速获取并解析出详细的中文地址（如“xx区 xx街道 xx大厦”）。

### 3. 📱 UI 安全区域适配 (Safe Area)
**目标**：确保内容不被刘海屏、挖孔或底部手势条遮挡。
*   **检查点**：
    *   **顶部**：首页的 "早安/晚安" 标题是否与状态栏有合适间距，不被摄像头遮挡。
    *   **底部**：记录页（RecordMood）底部的「保存记录」按钮是否悬浮在底部手势条之上，容易点击。
    *   **导航栏**：各个页面的顶部返回按钮是否因刘海屏而无法点击。

### 4. 🏷️ 标签管理入口 (Tag Management)
**目标**：验证从记录页直接跳转管理标签的功能。
*   **操作**：
    1.  进入「记录心情」页面。
    2.  找到 "选择情绪标签" 标题右侧的 **"管理"** 按钮。
    3.  点击它。
*   **预期**：直接跳转到标签管理页面，可以添加或删除标签，完成后返回记录页标签列表应更新。

### 5. 🖼️ 头像修改 (Avatar Change)
**目标**：验证头像修改功能是否正常工作。
*   **操作**：
    1.  进入「设置」 -> 「个人资料」。
    2.  点击头像或编辑图标。
    3.  选择「从相册选择」或「拍照」。
*   **预期**：
    *   应弹出系统权限请求（如首次）。
    *   选择图片后，可以进行裁剪（如果原生支持）。
    *   保存后，首页和侧边栏的头像应立即更新。

## 🚀 性能与优化建议 (Next Steps)

在验证完上述修复后，我们识别出以下进一步优化的机会：

1.  **📊 统计图表优化**
    *   **现状**：统计页面的 "趋势数据" 计算会加载所有历史记录，随着数据量增加可能会变慢。
    *   **建议**：在 Service 层增加按日期范围筛选的 API，或在 SQLite 层进行聚合查询，减少前端内存占用。

2.  **⚡ 图片存储优化**
    *   **现状**：应用保存的是原始或简单压缩的图片 Base64/DataURL。
    *   **建议**：在 Native 端引入更高级的图片压缩库，保存为 WebP 格式并存储到文件系统（而非数据库），仅在数据库存路径，以减小数据库体积。

3.  **🔔 交互反馈升级**
    *   **现状**：部分操作使用 `alert`/`confirm`。
    *   **建议**：引入 Toast 提示组件（轻量级消息条），提供更现代化、不打断用户的操作反馈。
